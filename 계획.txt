# Comment Tracker & Analytics — Development Specification

## 1. Project Overview

### 1.1 Purpose

A local DB-backed tool that accumulates client comment history across all projects, provides pattern analysis, tracks improvement trends, and connects to Lessons & Learned (L&L) workflows. The tool receives structured comment data (formatted by a Claude skill from raw client comment files) and stores, queries, and reports on it.

### 1.2 Problem Statement

- Client comments are managed per-project in isolated Excel files — no cross-project visibility.
- Recurring comment patterns (e.g., "Figure resolution too low") appear across projects without systematic detection.
- No way to measure improvement over time (e.g., "Are we getting fewer Minor-FigTable comments?").
- Past answers to similar comments cannot be searched or reused efficiently.
- L&L sessions lack data-driven evidence of recurring issues.

### 1.3 System Architecture: Two-Part Design

```
┌──────────────────────────────────────────────────────────┐
│  Part A: Claude Skill (runs in Claude chat interface)    │
│                                                          │
│  User uploads raw comment file (Excel/PDF/DOCX)         │
│  → Claude parses, classifies, and formats                │
│  → Outputs structured JSON/CSV per the Ingestion Schema  │
└──────────────────────┬───────────────────────────────────┘
                       │  structured file
                       ▼
┌──────────────────────────────────────────────────────────┐
│  Part B: Comment Tracker Program (built by Claude Code)  │
│                                                          │
│  CLI tool that:                                          │
│  - Imports structured comment data into SQLite DB        │
│  - Queries and searches past comments                    │
│  - Generates analytics and trend reports                 │
│  - Exports L&L-ready summaries                           │
│  - Tracks BSC performance metrics                        │
└──────────────────────────────────────────────────────────┘
```

**Why two parts?**
- Raw comment files from clients come in wildly different formats (Excel tables, PDF annotations, DOCX tracked changes, email text). Standardizing these requires LLM interpretation — best handled by Claude in chat.
- The program itself stays deterministic: it only accepts well-structured input and focuses on storage, querying, and analytics.

## 2. Data Model

### 2.1 Existing Classification System (from comment-response-guide skill)

The program must align with GeoView's established comment taxonomy:

**Severity:**
- `Major` — Impacts technical accuracy, requires reprocessing
- `Minor` — Cosmetic/editorial, no reprocessing needed

**Minor Sub-Categories (5-way):**
- `Typo` — Spelling, grammar, wording
- `Readability` — Sentence clarity, paragraph flow
- `FigTable` — Figure/table quality, captions, resolution
- `Format` — Style consistency, terminology, numbering
- `Reference` — TOC, cross-references, appendix links

**Status:**
- `Accepted` — Will fix as requested
- `Accepted (modified)` — Will fix but differently than requested
- `Noted` — Acknowledged, no change made (with justification)
- `Rejected` — Will not change (with justification)

**Exclusion:**
- Comments about submission/revision logistics, scope questions, timelines are excluded from performance metrics.

### 2.2 Database Schema (SQLite)

```sql
-- Projects table
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_code TEXT UNIQUE NOT NULL,       -- e.g., "JAKO2025"
    project_name TEXT NOT NULL,              -- e.g., "JAKO Route Survey 2025"
    client TEXT NOT NULL,                    -- e.g., "JAKO"
    report_type TEXT,                        -- Processing / Interpretation / Field / Other
    survey_type TEXT,                        -- OWF / Route / Geotech / Other
    start_date TEXT,                         -- YYYY-MM-DD
    end_date TEXT,
    notes TEXT,
    created_at TEXT DEFAULT (datetime('now'))
);

-- Comment batches (one per review cycle / revision)
CREATE TABLE batches (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL REFERENCES projects(id),
    revision TEXT NOT NULL,                  -- "Rev01", "Rev02", etc.
    reviewer TEXT,                           -- Client reviewer name/team
    received_date TEXT,                      -- YYYY-MM-DD
    source_file TEXT,                        -- Original filename for traceability
    total_comments INTEGER,                  -- Auto-calculated on import
    notes TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    UNIQUE(project_id, revision)
);

-- Individual comments
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    batch_id INTEGER NOT NULL REFERENCES batches(id),
    comment_number INTEGER NOT NULL,         -- Sequential # within batch
    section TEXT,                            -- "3.2, p.15", "Table 4.1", "Fig 5.3"
    comment_text TEXT NOT NULL,              -- Original comment (full text)
    severity TEXT NOT NULL CHECK(severity IN ('Major', 'Minor')),
    category TEXT NOT NULL,                  -- Major: "Technical"
                                             -- Minor: "Typo", "Readability", "FigTable",
                                             --        "Format", "Reference"
    status TEXT NOT NULL CHECK(status IN ('Accepted', 'Accepted (modified)', 'Noted', 'Rejected')),
    response_text TEXT,                      -- Our answer
    assignee TEXT,                           -- Person who resolved
    resolved_date TEXT,
    excluded INTEGER DEFAULT 0,             -- 1 = excluded from stats (logistics/scope)
    exclude_reason TEXT,                     -- Why excluded
    confidence TEXT DEFAULT 'High',          -- Classification confidence: High/Medium/Low
    tags TEXT,                               -- Comma-separated free tags for custom grouping
    created_at TEXT DEFAULT (datetime('now'))
);

-- L&L links (comments flagged as lessons learned)
CREATE TABLE ll_flags (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    comment_id INTEGER NOT NULL REFERENCES comments(id),
    ll_type TEXT NOT NULL CHECK(ll_type IN ('Recurring', 'Process', 'Prevention', 'Improvement')),
    ll_summary TEXT,                         -- One-line lesson summary
    ll_action TEXT,                          -- Suggested preventive action
    flagged_date TEXT DEFAULT (date('now')),
    flagged_by TEXT DEFAULT 'system'         -- 'system' (auto) or 'manual'
);

-- Full-text search virtual table
CREATE VIRTUAL TABLE comments_fts USING fts5(
    comment_text,
    response_text,
    section,
    tags,
    content='comments',
    content_rowid='id'
);
```

## 3. Program Features

### 3.1 Import

```bash
# Import from JSON (includes project + batch metadata)
comment-tracker import --file comments_JAKO2025_Rev02.json

# Import from CSV (metadata via CLI)
comment-tracker import \
  --file comments.csv \
  --project-code JAKO2025 \
  --project-name "JAKO Route Survey 2025" \
  --client JAKO \
  --report-type Processing \
  --revision Rev02 \
  --reviewer "Client Technical Team"

# Re-import (update existing batch — warns before overwrite)
comment-tracker import --file comments_JAKO2025_Rev02.json --update
```

**Import behavior:**
- If project doesn't exist, create it.
- If project exists, reuse it (match by `project_code`).
- If batch (project + revision) already exists, refuse unless `--update` flag is set.
- Validate all fields against schema before inserting (fail fast with clear error).
- Update FTS index after import.
- Print summary: "Imported 23 comments (5 Major, 18 Minor) for JAKO2025 Rev02."

### 3.2 Search & Query

```bash
# Full-text search across all comments and responses
comment-tracker search "figure resolution"

# Search with filters
comment-tracker search "velocity" --client JAKO --severity Major
comment-tracker search "coordinate" --category FigTable --status Accepted

# List all comments for a project
comment-tracker list --project JAKO2025

# List comments for a specific revision
comment-tracker list --project JAKO2025 --revision Rev02

# Show specific comment detail (with full response)
comment-tracker show --id 42

# Find similar past responses (for reuse)
comment-tracker similar "Please improve figure resolution to 300 dpi"
```

**`similar` command:**
- Uses FTS5 to find comments with matching terms.
- Returns top N results ranked by relevance.
- Shows: project, revision, original comment, response, severity, category.
- Purpose: When writing a new response, find how similar comments were answered before.

### 3.3 Analytics & Statistics

```bash
# Project-level summary
comment-tracker stats --project JAKO2025

# Client-level summary (across all projects for that client)
comment-tracker stats --client JAKO

comment-tracker stats

comment-tracker trend --project JAKO2025

# Category distribution report
```

### 3.4 L&L Integration

#### 3.4.1 Auto-Flagging

The program identifies comments that may warrant L&L documentation:

```bash
# Find recurring patterns across projects (potential L&L candidates)
comment-tracker ll-scan

# Flag a specific comment as L&L material
comment-tracker ll-flag --id 42 --type Recurring --summary "Figure resolution consistently below 300 dpi"

# View all L&L-flagged items
comment-tracker ll-list

# Export L&L summary for a session
comment-tracker ll-export --output ll_input.json
```

**Auto-scan rules (`ll-scan`):**
- **Recurring:** Same category + similar text appears in 3+ different projects → flag as `Recurring`
- **Process:** Major comments that caused reprocessing → flag as `Process`
- **Prevention:** Comments that could have been prevented by a checklist → flag as `Prevention`

## 6. Claude Skill Specification (Part A)

### 6.1 Skill Purpose

The Claude skill (used in a regular Claude chat, NOT in the program) transforms raw client comment files into the structured Ingestion Schema defined in Section 2.3.

### 6.2 Skill Trigger

User uploads a raw comment file and says something like:
- "Comment Tracker 포맷으로 정리해줘"
- "이 코멘트 파일 정리"
- "코멘트 DB용으로 변환"

### 6.3 Skill Workflow

```
1. User uploads raw comment file (Excel/PDF/DOCX/email text)
2. Claude reads the file and identifies:
   - Comment list (what the client wrote)
   - Section/location references
   - Any existing severity markings by client
3. Claude asks user for missing metadata:
   - Project code, client, report type (if not obvious)
   - Revision number
   - Reviewer name
4. Claude classifies each comment:
   - Apply severity (Major/Minor) using classification-rules.md
   - Apply Minor sub-category (5-way) using keyword matching
   - Apply exclusion rules
   - Mark confidence level
5. Claude generates the structured output:
   - JSON file following Ingestion Schema
   - Summary statistics
6. User reviews, adjusts if needed
7. User saves JSON and feeds to: comment-tracker import --file <file>
```

### 6.4 Classification Logic (for the skill)

The skill must follow the existing classification-rules.md:

**Priority order:**
1. Check exclusion rules → if excluded, set `excluded: true` with reason
2. Check if Major (technical impact, reprocessing needed) → severity: "Major", category: "Technical"
3. For Minor, classify by keyword:
   - Category 1 keywords → "Typo"
   - Category 3 keywords → "FigTable" (check before Cat 2)
   - Category 5 keywords → "Reference"
   - Category 4 keywords → "Format"
   - Default → "Readability" (catch-all)

**Confidence assignment:**
- High: Unambiguous keyword match
- Medium: Could belong to two categories
- Low: No clear match, defaulted to Readability

### 6.5 Skill Output Requirements

The skill must produce:
1. **JSON file** exactly matching the Ingestion Schema (Section 2.3)
2. **Console summary** showing:
   - Total comments parsed
   - Major/Minor split
   - Category distribution
   - Any excluded comments with reasons
   - Any Low-confidence classifications for user review
3. **Review prompt**: "Low-confidence 분류 3건이 있습니다. 확인해주세요:" followed by the ambiguous items

## 10. Recurring Theme Detection Algorithm

### 10.1 Logic

```python
def find_recurring_themes(min_occurrences=3, min_projects=2):
    """
    Find comment themes that appear across multiple projects.

    Approach:
    1. Group comments by category
    2. Within each category, extract key terms from comment_text
    3. Cluster similar comments using term overlap
    4. Flag clusters that span >= min_projects and have >= min_occurrences
    """

    # Step 1: Extract significant terms (nouns, technical terms)
    # Use simple word frequency — no ML needed
    # Exclude stop words and common report terms

    # Step 2: For each term, count:
    #   - How many comments contain it
    #   - How many unique projects
    #   - How many unique clients

    # Step 3: Filter by thresholds
    #   - occurrences >= min_occurrences
    #   - unique projects >= min_projects

    # Step 4: Group and rank by occurrence count
```

### 10.2 Term Extraction

Simple keyword-based approach (no ML):
- Split comment text into words
- Lowercase, strip punctuation
- Remove stop words (English + domain-specific: "please", "should", "the", etc.)
- Keep 2-grams for phrases: "figure resolution", "coordinate system", "sea bed"
- Count term frequency across all comments

### 10.3 Similarity for `similar` Command

Uses SQLite FTS5 `rank` function:
```sql
SELECT c.*, rank
FROM comments_fts
JOIN comments c ON comments_fts.rowid = c.id
WHERE comments_fts MATCH ?
ORDER BY rank
LIMIT 10;
```

## 12. Integration Points

### 12.1 With comment-response-guide Skill

The existing skill classifies and generates answer sheets. The Comment Tracker extends this by:
- Accumulating answer sheets over time
- Providing historical context ("last time this client asked about X, we answered Y")
- Feeding back statistics to improve future answer quality

### 12.2 With lessons-learned-writer Skill

The L&L export (`ll-export`) produces structured data that the L&L skill can consume:
- Recurring patterns become L&L candidates
- Evidence (occurrence count, affected projects) supports L&L documentation
- Suggested actions come pre-drafted from the scan

### 12.3 With BSC Daily Report Skill

The `bsc` command output can feed into the BSC monthly performance documentation:
- Comment reduction percentages as performance evidence
- Category trends showing improvement areas

### 12.4 Workflow Summary

```
Raw comment file
    │
    ├──→ [Claude Chat] comment-response-guide skill
    │        → Answer Sheet for client submission
    │
    ├──→ [Claude Chat] Comment Tracker formatting skill
    │        → Structured JSON
    │            │
    │            ▼
    │    [CLI] comment-tracker import
    │            │
    │            ▼
    │    SQLite DB (accumulated history)
    │            │
    │            ├──→ comment-tracker stats/trend → BSC report input
    │            ├──→ comment-tracker ll-scan → L&L session input
    │            ├──→ comment-tracker similar → Response reuse
    │            └──→ comment-tracker search → Ad-hoc queries
    │
    └──→ [When L&L session] lessons-learned-writer skill
             ← consumes ll-export output
```

여기선 CLI 방식으로 되어있는데, 사용자 친화적인 GUI 방식으로 제작해줘. 전문적인 형태로 부탁함. 프로그램 만들고 나서 클로드에게 새로운 채팅방 만들어서 스킬 알려주고, 그 스킬대로 이 프로그램에 입력시킬 포맷의 코멘트 관련 데이터를 제작할것임.
